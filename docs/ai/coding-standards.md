## Цели стандартов

- Минимизировать риск регрессий при использовании AI‑агента;
- обеспечить единый стиль кода в C‑клиенте и Java‑сервере;
- сформировать понятные требования к тестам и проверкам.

## Общие правила для кода

- **Следовать существующему стилю файла.** AI обязан ориентироваться на форматирование и приёмы, уже используемые в конкретном файле.
- **Малые, изолированные изменения.** Избегать массовых автоматических рефакторингов без прямого запроса.
- **Чёткие комментарии.** При изменении поведения обновлять или добавлять комментарии.
- **Заголовок файла.** В начале нового/отредактированного файла добавлять или обновлять комментарий с назначением:
  - C:

```c
/* Purpose: <краткое назначение файла>. Maintainer: AI-assisted change. */
```

  - Java:

```java
// Purpose: <краткое назначение файла>. Maintainer: AI-assisted change.
```

  - Markdown:

```markdown
<!-- Purpose: <краткое назначение файла>. Generated/edited with AI assistance. -->
```

## Стиль C‑кода (`app/`)

- Отступы: использовать тот же стиль, что и в файле (обычно 4 пробела, без табов).
- Ширина строки: стараться не превышать ~100 символов.
- Именование:
  - функции и переменные — в стиле, принятом в модуле (snake_case и т.п.);
  - избегать неочевидных сокращений.
- Управление ресурсами:
  - при добавлении кода, выделяющего ресурсы (malloc, fopen, сокеты), обязательно описывать путь их освобождения;
  - при ошибках аккуратно освобождать уже выделенные ресурсы.
- Обработка ошибок:
  - всегда проверять коды возврата системных вызовов и библиотечных функций;
  - при ошибках логировать достаточный контекст через существующие механизмы логирования.

## Стиль Java‑кода (`server/src/main/java`)

- Отступы: как в исходном файле (2 или 4 пробела).
- Именование:
  - классы — `CamelCase`;
  - методы и поля — `camelCase`, отражают назначение.
- Android‑совместимость:
  - учитывать проверки из `AndroidVersions.java`;
  - не использовать новые API без соответствующих проверок и fallback‑поведения.
- Исключения:
  - не глушить исключения без логирования;
  - при перехвате исключения указывать, почему оно может происходить и что делать пользователю/разработчику.

## Стиль документации (`doc/` и `docs/ai/`)

- Язык: кратко, по делу, без «воды».
- Всегда приводить рабочие примеры команд:

```bash
scrcpy --video-codec=h265 --max-fps=60
```

- Для новых разделов указывать, на какие опции/файлы ссылается документация.

## Требования к тестам

- **При любом изменении логики**:
  - указать, какие тесты нужно запустить (конкретные команды);
  - по возможности добавить/обновить тесты.

- **Java‑сервер**:

```bash
cd server
./gradlew test
```

- **C‑клиент** (пример для Meson/Ninja):

```bash
meson setup builddir
ninja -C builddir
```

- При изменении CLI‑опций или протокола — описать ручные шаги проверки (пример команды `scrcpy`, ожидаемый результат).

## Формат коммитов

- Все AI‑коммиты должны начинаться с префикса:

```text
[AI] <краткое описание>
```

Примеры:

```text
[AI] Fix resource leak in audio capture
[AI] Add CLI option for max bitrate
```

## Защищённые участки кода

- Код между маркерами:

```c
/* AI:DO_NOT_EDIT-START */
... защищённый код ...
/* AI:DO_NOT_EDIT-END */
```

или аналогичными комментариями в других языках не должен изменяться AI‑агентом.
